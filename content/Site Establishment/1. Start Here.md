## Foundation and Terminology:
### What is **node.js**?
The over-simplified version as it relates to **Quartz** and **Obsidian**:
- **Quartz**, uses node.js to execute JavaScript on the 'backend' via terminal commands on your device to generate static-site information from Markdown files (Obsidian's bread and butter).
- **JavaScript** natively runs in an internet browser environment (often referred to as the 'frontend') which can be limiting for certain use cases.
- [node.js](https://nodejs.org/en/about) is a runtime which facilitates the execution of JavaScript code directly on your device. Without **node.js**, **Quartz** couldn't transform your precious local markdown files into a magnificent personalized website.

A vastly more technical explanation of **Quartz** architecture can be found [here](https://quartz.jzhao.xyz/advanced/architecture)

### What is **Node Version Manager (NVM)**?
Now that we've established the crucial importance of **node.js**, we need to manage it:
- As you may have deduced, **node.js** was *NOT* developed solely for **Quartz**'s implementation.
- **Quartz**, as well as other projects, are built with respect to specific versions of **node.js**.
- One does not simply update, or roll-back, a singular installation of **node.js** without the risk of breaking a program which depends on the original version.
- This is where **NVM** comes in:
	- It allows you to install multiple versions of **node.js**
	- It also facilitates switching between those versions via terminal command line instructions.

## Installing the Prerequisites:


>- *curl -o-* :  downloads an installation script from the provided URL and outputs it in a standard manner.
>- | (*vertical pipe symbol*): This output is then sent to *bash* to be executed.

3. Create/edit a *.zshrc* file or *.bashrc* file, or *.bash_profile* file:
	- *First:* Check if one of these file's already exists.
	- The following code will list the files in your current directory:
	

>[!example]- Breaking Down the Code:
>- *ls*: lists the contents of the current directory.
>- *-a*: includes hidden files in the list (anything that begins with a *.* is hidden).
>- *-r*: prints the list in reverse order (the files we're looking for will be near the bottom this way). 

- Depending on your default terminal path from [[1. Start Here#^c48658|Step 2]], follow the applicable steps:
>[!check]- */bin/zsh*: if your default terminal uses *zsh*(*Z Shell*):
>- if you do *NOT* have a *.zshrc* file in your home directory, enter the following into your
> terminal:
>```bash
>touch ~/.zshrc
>```
>>[!example]- Breaking Down the Code:
>>- *touch*: create file with designated name (*.zshrc* in this case )
>>- *~/*: ensures the file is created in your home directory (not strictly necessary if you're already in the home directory)
>
>- if you just created or already have a *.zshrc* file in your home directory, enter the following into your terminal:
>```bash
>nano .zshrc
>``` 
>>[!example]- Breaking Down the Code:
>>- *nano*: opens a text editor within the terminal on the file specified
>- copy and paste the following code block into the terminal text editor:
>```bash
>export NVM_DIR="$([ -z "${XDG_CONFIG_HOME-}" ] && printf %s "${HOME}/.nvm" || printf %s "${XDG_CONFIG_HOME}/nvm")"
[ -s "$NVM_DIR/nvm.sh" ] && . "$NVM_DIR/nvm.sh"
[ -s "$NVM_DIR/bash_completion" ] && . "$NVM_DIR/bash_completion"
>``` 
>>[!example]- Breaking Down the Code:
>>-Simply: the next command checks to see if the correct environment variable is set for **NVM** and expands the node version manager functionality to the appropriate (current) directory if necessary. This ensures 'nvm' instructions will run correctly in the terminal.
>>-Why is the snippet I provided different from the snippet suggested by the **NVM** installer ?
>>	- *Either snippet can be used*, but the snippet I provided in the code block offers more flexibility as it standardizes where configuration files are located in respect to different applications.
>>	- Use the image snippet if you *ALWAYS* want NVM to be installed in the default location.
>>	- Use the snippet provided in the code block if you'd rather respect the `XDG_CONFIG_HOME` environment variable(this should result in a more standardized configuration directory).
>
